# CEO Plan Review — Plinko Physics Delay VST

Generated by /plan-ceo-review on 2026-06-17 · Mode: SELECTIVE EXPANSION · Stack: JUCE/C++

## Concept

A visual audio effect VST. A ball drops through a user-built Plinko board; each peg hit
becomes a delay tap whose timing/pan/pitch/filter/level is a function of the peg's
position. The ball loops forever (guaranteed never stuck). Prototype-then-build in JUCE.

## Market context (why this can win)

The "bouncing ball delay" is proven and shipping — Sinevibes Dispersion and Smart
Electronix Bouncy both exist, and Dispersion confirms the rhythmic, tempo-synced
multi-tap interpretation is the musical one. But those use the bounce as a *math
metaphor*. None ship an actual interactive 2D physics playfield with user-placed pegs.
**That interactive, screenshot-able board is the differentiation, not the delay math.**

## Key reframe

The strong musical interpretation is NOT "bounces vaguely change reverb depth." It is:
**each peg hit = a discrete delay tap; the peg's position sets that tap's character.**
The ball's path literally draws a rhythm of echoes across the stereo field. More musical,
more controllable, more legible than continuous knob-wobble.

## Decisions made (this review)

| # | Decision | Choice | Why |
|---|---|---|---|
| 1 | Prototype path | Thin sketch, build real early | Product is GUI-forward; Python can't model the hard parts |
| 2 | Stack | JUCE / C++ | Mature custom-GUI + all plugin formats + biggest tutorial base |
| 3 | Review mode | Selective Expansion | Hold a tight shippable core; cherry-pick delights consciously |
| 4 | Physics architecture | Deterministic, audio-time-driven (day 1) | Render parity + reload reproducibility; cannot be retrofitted cheaply |
| 5 | No-stuck mechanism | Layered: nudge + timeout backstop | Organic motion + mathematically guaranteed exit |
| 6 | Drive 3rd-party VSTs | Built-in effects only for v1 (seam for v2) | Hosting is a multi-month separate product; don't sink the first ship |
| 7 | Determinism promise | Same-binary reproducibility only | Box2D float math isn't bit-identical cross-machine; this covers what musicians need |
| 8 | Threading | Physics on audio thread, hardened | Async worker breaks offline-render parity; preallocate + caps-to-budget + no watchdog |
| 9 | Balls in v1 | One ball | One ball fully expresses the idea; multi-ball multiplies the hard problems |
| 10 | Sound mapping | Specified concretely in ARCHITECTURE.md §5 | The peg→sound mapping IS the product; build from a real hypothesis, not a footnote |

## Accepted scope (v1)

Baseline (bulletproofed): ONE ball, draggable pegs + walls, gravity/ball-size/drop-point,
tempo-synced multi-tap delay, live visual board, guaranteed no-stuck, same-binary
deterministic physics, preset versioning.

Accepted additions: savable preset boards · per-peg pitch + filter · reverb send ·
deterministic replay + debug overlay · extra shapes · internal event-queue seam (for v2).

## Deferred (see BACKLOG.md)

- Multiple simultaneous balls — moved out of v1 (multiplies determinism/CPU cost).
- Modulation-out (MIDI/MPE/host routing) — v2, via the event queue.
- Full nested VST hosting — v2/v3, separate product.
- Cross-machine bit-identity (fixed-point physics) — research-grade, only if needed.
- Faithful Python DSP prototype — intentionally skipped (mapping specified in ARCHITECTURE.md instead).

## Outside-voice review (absorbed)

An independent reviewer challenged the locked plan. Findings accepted and folded into
ARCHITECTURE.md:
- Cross-machine determinism with Box2D is not real → promise same-binary reproducibility only (§2).
- "Cheap" physics ≠ "bounded" → harden the on-thread design: preallocate, caps sized by
  profiling, remove the watchdog (a watchdog that fires IS the glitch) (§3).
- Cut to one ball for v1 (§12).
- The peg→sound mapping is the actual product → specified concretely (§5).
- VST3 can't hold ~640 peg params → board geometry is ValueTree state, ~12-16 automatable
  macros (§6). **Resolve before building the editor.**
- Pitch-shift latency/PDC and sample-rate changes flagged (§8).

Disagreement: the reviewer wanted physics moved off the audio thread to a worker; rejected
because an async worker can't stay sample-synced during offline render. Kept on-thread but
hardened instead.

**SUPERSEDED by eng review 2026-06-17:** the "same-binary only" determinism call (tension #1)
was based on the outside voice's claim that Box2D isn't cross-platform deterministic. That is
false for **Box2D v3**, which was engineered for cross-platform determinism. Promise upgraded
to full cross-platform identity. See ARCHITECTURE.md §2 and ENG-LAYOUT.md §6.

## The two silent killers (must handle, easy to miss)

1. **Denormals** — reverb tails on silence cause a CPU storm that feels like a DAW freeze.
   Fix: FTZ/DAZ flags.
2. **NaN propagation** — one bad physics frame becomes permanent loud DC or silence. Fix:
   finite-check every substep, reset sim to last good seeded state.

## Reviewer's bottom line

The concept is sound and market-validated. There was exactly one foundational decision
(deterministic, audio-time-driven physics) that everything else hangs off, and it's
resolved. Scope is disciplined for a first VST: the dream (drive any plugin) is preserved
as a clean v2 via the event-stream seam rather than a rewrite. Biggest ongoing risk is
scope creep back toward hosting before v1 ships — hold the line.
